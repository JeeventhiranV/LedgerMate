Q1
How does ConcurrentHashMap handle thread safety without full locking?
Hard🔥 Hot
CAS + Fine-Grained Locking (Java 8+):
ConcurrentHashMap uses CAS (Compare-And-Swap) for atomic operations and synchronized blocks only at the bucket level, not the entire map.

Key mechanisms:
  • volatile Node[] array — ensures visibility across threads
  • On empty bucket: uses CAS to insert — no lock needed
  • On occupied bucket: synchronized(head) — only locks that bucket
  • Supports 16 concurrent segments by default → 16x parallelism vs Hashtable
  • Java 8 replaced segments with node-level locking entirely
🎯 Interview TipContrast with Hashtable which locks the entire object on every operation — even reads.
import java.util.concurrent.*;

public class ConcurrentHashMapDemo {
  public static void main(String[] args) throws InterruptedException {
    ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

    // ✅ putIfAbsent — atomic check-then-act, no race condition
    map.putIfAbsent("counter", 0);

    // ✅ compute — atomically update a value
    map.compute("counter", (k, v) -> v == null ? 1 : v + 1);

    // Demonstrate concurrent writes from 10 threads
    ExecutorService pool = Executors.newFixedThreadPool(10);
    for (int i = 0; i < 10; i++) {
      pool.submit(() -> {
        // compute is bucket-level synchronized — safe for concurrent update
        map.compute("counter", (k, v) -> v + 1);
      });
    }
    pool.shutdown();
    pool.awaitTermination(5, TimeUnit.SECONDS);
    System.out.println("Counter: " + map.get("counter")); // Always 11 — no race

    // ❌ NOT safe — get+put is not atomic:
    // map.put("x", map.getOrDefault("x", 0) + 1);  // race condition!

    // ✅ SAFE — merge atomically adds value:
    map.merge("x", 1, Integer::sum); // if absent: set 1; else: add 1

    // ConcurrentHashMap vs Hashtable vs synchronizedMap
    // Hashtable: synchronized on 'this' for every op — no concurrency
    // Collections.synchronizedMap: wraps map, same lock for all ops
    // ConcurrentHashMap: bucket-level lock + CAS — true concurrency
  }
}
Q2
How does HashMap work internally — before and after Java 8?
Medium🔥 Hot
Before Java 8: Array of LinkedList buckets. Collision = O(n) worst case.

Java 8+:
  • Array + LinkedList → converts to TreeMap (Red-Black Tree) when bucket length ≥ 8 (treeification)
  • Converts back to list when size drops below 6
  • Threshold: capacity × loadFactor (default 16 × 0.75 = 12)
put() flow: compute hash → find bucket index → check equals() → insert or replace → resize if threshold exceeded

What is Rehashing? When size exceeds threshold, a new array of 2× capacity is created and all entries are re-inserted using updated hash % newCapacity. 
import java.util.*;

public class HashMapInternalsDemo {
  public static void main(String[] args) {
    // Default capacity=16, loadFactor=0.75 → resize at 12 entries
    Map<String, Integer> map = new HashMap<>();

    // put() flow: hash(key) → index = hash & (n-1) → insert Node
    map.put("alice", 1); // bucket index = "alice".hashCode() & 15
    map.put("bob", 2);
    map.put("carol", 3);

    // Collision demo — keys with same hash go to same bucket
    // "FB" and "Ea" have same hashCode in Java (collision):
    Map<String, Integer> collision = new HashMap<>();
    collision.put("FB", 1); // "FB".hashCode() == "Ea".hashCode() == 2236
    collision.put("Ea", 2); // stored in same bucket as linked list

    // Java 8+: bucket converts to Red-Black Tree when ≥ 8 entries in bucket
    // (treeification threshold = 8, untreeify threshold = 6)
    // This changes worst-case bucket lookup from O(n) → O(log n)

    // Specify initial capacity to avoid rehashing for known sizes:
    int expectedEntries = 1000;
    Map<String, String> optimized = new HashMap<>(
      (int) (expectedEntries / 0.75) + 1 // initialCapacity to hold 1000 without resize
    );

    // ❌ Never use mutable objects as keys — hashCode changes after mutation
    // ✅ Keys should be immutable (String, Integer, UUID are ideal)
    System.out.println(map.get("alice")); // 1
    System.out.println(collision.get("Ea")); // 2 — equals() resolves collision
  }
}
Q3
What is the Java Memory Model (JMM)? Explain volatile and happens-before.
Hard
JMM defines rules for how threads interact through memory — visibility and ordering guarantees.

volatile:
  • Guarantees visibility: writes visible to all threads immediately
  • Prevents CPU reordering of reads/writes around the variable
  • Does NOT guarantee atomicity — i++ on volatile is still not thread-safe
happens-before: If action A happens-before B, A's effects are visible to B. Examples:
  • Thread.start() happens-before any action in the started thread
  • synchronized unlock happens-before subsequent lock by another thread
  • Write to volatile happens-before subsequent read of same field
// ❌ VISIBILITY BUG — without volatile, thread may never see updated flag
public class VisibilityBugDemo {
  private static boolean running = true; // ❌ not volatile — CPU may cache it

  public static void main(String[] args) throws InterruptedException {
    Thread worker = new Thread(() -> {
      while (running) { // may loop forever — reads stale cached value!
        // do work...
      }
      System.out.println("Stopped");
    });
    worker.start();
    Thread.sleep(100);
    running = false; // main thread writes — worker may not see it!
  }
}

// ✅ FIX — volatile guarantees write is flushed to main memory immediately
public class VolatileFixDemo {
  private static volatile boolean running = true; // ✅ volatile

  public static void main(String[] args) throws InterruptedException {
    Thread worker = new Thread(() -> {
      while (running) { // always reads from main memory — sees update
        // do work...
      }
      System.out.println("Stopped cleanly"); // ✅ always prints
    });
    worker.start();
    Thread.sleep(100);
    running = false; // guaranteed visible to worker thread
    worker.join();
  }
}

volatile int counter = 0;
counter++; // still 3 steps (read → increment → write) — not atomic!

// ✅ Use AtomicInteger for atomic compound ops:

Q5
Explain Garbage Collection internals — Minor GC, Major GC, Full GC, and G1 GC.
Hard🔥 Hot
Heap Generations:
  • Young Gen (Eden + S0 + S1): Most objects die here. Minor GC is fast (stop-the-world, milliseconds)
  • Old Gen (Tenured): Long-lived objects. Major GC is slower
  • Full GC: Cleans entire heap — most expensive, should be rare
Object Promotion: After surviving N GC cycles (default 15 for G1), object moves to Old Gen.

G1 GC (Java 9+ default):
  • Divides heap into equal-sized regions (~2000 of them)
  • Each region can be Eden, Survivor, or Old at any time
  • Prioritizes collecting regions with most garbage (Garbage First)
  • Concurrent marking — minimizes stop-the-world pauses
  • ZGC/Shenandoah: ultra-low latency, sub-millisecond pauses
Q6
What causes memory leaks in Java despite GC? How to detect and fix?
Hard🏭 Prod
Common causes:
  • Static references to large objects — GC can't collect
  • Unclosed resources (JDBC connections, streams) — resource leak
  • Listeners/callbacks not de-registered
  • Overriding equals() but not hashCode() → duplicate HashMap entries
  • ThreadLocal not removed after task completes (thread pool reuse)
  • Inner class holding reference to outer class
Detection:
  • jmap -histo <pid> — heap histogram
  • jvisualvm / JProfiler — heap dump analysis
  • -Xmx -verbose:gc flags for GC monitoring
  • Heap dump on OOM: -XX:+HeapDumpOnOutOfMemoryError
// ❌ LEAK 1: ThreadLocal not removed in thread pool (threads are reused)
static ThreadLocal<UserContext> userCtx = new ThreadLocal<>();

// In a servlet/request handler:
public void handleRequest(HttpRequest req) {
  userCtx.set(new UserContext(req.getUserId())); // ❌ set but never removed
  processRequest();
  // Thread returns to pool with UserContext still set → LEAK
}

// ✅ FIX — always remove ThreadLocal in finally:
public void handleRequestFixed(HttpRequest req) {
  try {
    userCtx.set(new UserContext(req.getUserId()));
    processRequest();
  } finally {
    userCtx.remove(); // ✅ always clean up, even on exception
  }
}

Q8
What is the difference between fail-fast and fail-safe iterators?
Medium
Fail-Fast: Throws ConcurrentModificationException if collection is modified during iteration. Works on internal modCount. Examples: ArrayList, HashMap iterators.

Fail-Safe: Iterates over a copy of the collection — no exception if modified. Examples: CopyOnWriteArrayList, ConcurrentHashMap iterators.

PitfallFail-safe iterators may not reflect latest changes since they work on a snapshot. Higher memory usage.
import java.util.*;
import java.util.concurrent.*;

public class IteratorDemo {
  public static void main(String[] args) {

    // ❌ FAIL-FAST — ConcurrentModificationException
    List<String> list = new ArrayList<>(List.of("a", "b", "c"));
    try {
      for (String s : list) {
        if (s.equals("b")) {
          list.remove(s); // ❌ modCount changed → ConcurrentModificationException
        }
      }
    } catch (ConcurrentModificationException e) {
      System.out.println("Fail-fast triggered: " + e.getClass().getSimpleName());
    }

    // ✅ FIX 1 — use Iterator.remove() to safely remove during iteration
    Iterator<String> it = list.iterator();
    while (it.hasNext()) {
      if (it.next().equals("a")) {
        it.remove(); // ✅ safe — updates modCount correctly
      }
    }

    // ✅ FIX 2 — CopyOnWriteArrayList (fail-safe, iterates snapshot)
    CopyOnWriteArrayList<String> safeList =
        new CopyOnWriteArrayList<>(List.of("x", "y", "z"));
    for (String s : safeList) {
      safeList.add("new"); // ✅ no exception — writes a fresh copy of array
      System.out.println(s); // prints x, y, z — snapshot, not modified view
    }
    System.out.println("Size after: " + safeList.size()); // 6 (3 added)

    // ConcurrentHashMap iterator is also fail-safe (weakly consistent)
    ConcurrentHashMap<String, Integer> chm = new ConcurrentHashMap<>();
    chm.put("a", 1); chm.put("b", 2);
    for (String key : chm.keySet()) {
      chm.put("c", 3); // ✅ no exception — may or may not see "c" in iteration
    }
  }
}
Q9
What happens during a deadlock? How would you debug it in production?
Hard🏭 Prod
Deadlock: Two or more threads waiting for each other's locks — circular dependency.

Four Conditions (Coffman): Mutual exclusion, Hold & Wait, No preemption, Circular wait.

Detection in production:
  • jstack <pid> — prints thread dump, highlights deadlock
  • VisualVM / JConsole → Thread tab → "Detect Deadlock" button
  • Look for threads in BLOCKED state waiting for each other
Prevention:
  • Always acquire locks in same order across threads
  • Use tryLock(timeout) from ReentrantLock
  • Use higher-level concurrency utilities (Executors, BlockingQueue)
import java.util.concurrent.locks.*;

// ❌ DEADLOCK — Thread A holds lock1, waits for lock2
//              Thread B holds lock2, waits for lock1 → circular wait
public class DeadlockDemo {
  private static final Object lock1 = new Object();
  private static final Object lock2 = new Object();

  public static void main(String[] args) {
    Thread t1 = new Thread(() -> {
      synchronized (lock1) {
        System.out.println("T1 acquired lock1, waiting for lock2...");
        try { Thread.sleep(50); } catch (InterruptedException e) {}
        synchronized (lock2) { // ❌ waits for T2 to release lock2
          System.out.println("T1 acquired lock2");
        }
      }
    }, "Thread-A");

    Thread t2 = new Thread(() -> {
      synchronized (lock2) {
        System.out.println("T2 acquired lock2, waiting for lock1...");
        try { Thread.sleep(50); } catch (InterruptedException e) {}
        synchronized (lock1) { // ❌ waits for T1 to release lock1 — DEADLOCK
          System.out.println("T2 acquired lock1");
        }
      }
    }, "Thread-B");

    t1.start(); t2.start(); // Both threads block forever
  }
}

// ✅ FIX 1 — always acquire locks in consistent order (lock1 before lock2)
synchronized (lock1) { synchronized (lock2) { /* work */ } } // both threads same order

// ✅ FIX 2 — tryLock with timeout (ReentrantLock) — breaks deadlock automatically
ReentrantLock rl1 = new ReentrantLock();
ReentrantLock rl2 = new ReentrantLock();

public void transferSafe() throws InterruptedException {
  while (true) {
    if (rl1.tryLock(100, java.util.concurrent.TimeUnit.MILLISECONDS)) {
      try {
        if (rl2.tryLock(100, java.util.concurrent.TimeUnit.MILLISECONDS)) {
          try {
            doTransfer(); // ✅ both locks acquired — safe to proceed
            return;
          } finally { rl2.unlock(); }
        }
      } finally { rl1.unlock(); }
    }
    // Failed to acquire both — back off and retry (avoids deadlock)
    Thread.sleep(10);
  }
}

Q10
What are SOLID principles? Give real-world Java examples.
Medium
  • Single Responsibility: A class should do one thing. OrderService handles orders, not email sending.
  • Open/Closed: Open for extension, closed for modification. Add new payment type via new class, not editing existing one.
  • Liskov Substitution: Subtypes must be substitutable for base type without breaking behavior.
  • Interface Segregation: Don't force clients to implement unused methods. Split fat interfaces.
  • Dependency Inversion: Depend on abstractions, not concretions. Inject PaymentGateway interface, not StripePayment directly.
// ── S: Single Responsibility Principle ─────────────────────────────────────
// ❌ Bad — one class handles orders AND email AND PDF
class OrderService {
  void placeOrder(Order o) { /* save */ }
  void sendConfirmationEmail(Order o) { /* SMTP */ }
  void generateInvoicePdf(Order o) { /* PDF */ }
}
// ✅ Good — each class has one reason to change
class OrderService   { void placeOrder(Order o)  { /* only order logic */ } }
class EmailService   { void sendEmail(Order o)   { /* only email logic */ } }
class InvoiceService { void generatePdf(Order o) { /* only PDF logic  */ } }

// ── O: Open/Closed Principle ────────────────────────────────────────────────
// ❌ Bad — must edit existing class to add new payment type
class PaymentProcessor {
  void process(String type) {
    if (type.equals("stripe"))  { /* ... */ }
    else if (type.equals("paypal")) { /* ... */ } // keep modifying ❌
  }
}
// ✅ Good — extend via new class, never touch existing code
interface PaymentGateway { void process(Order order); }
class StripeGateway  implements PaymentGateway { public void process(Order o) { /* Stripe API */ } }
class PayPalGateway  implements PaymentGateway { public void process(Order o) { /* PayPal API */ } }
class RazorpayGateway implements PaymentGateway { public void process(Order o) { /* new — no edits above */ } }

// ── L: Liskov Substitution Principle ───────────────────────────────────────
// ❌ Bad — Square breaks Rectangle contract (setWidth changes height too)
class Rectangle {
  int w, h;
  void setWidth(int w)  { this.w = w; }
  void setHeight(int h) { this.h = h; }
  int  area()            { return w * h; }
}
class Square extends Rectangle {
  void setWidth(int w)  { this.w = this.h = w; } // ❌ violates LSP
}
// ✅ Good — separate abstractions; use interface/composition
interface Shape { int area(); }
record Rectangle(int w, int h) implements Shape { public int area() { return w * h; } }
record Square(int side)       implements Shape { public int area() { return side * side; } }

// ── I: Interface Segregation Principle ─────────────────────────────────────
// ❌ Bad — fat interface forces implementors to stub unused methods
interface Worker {
  void work();
  void eat();   // ❌ robots don't eat — forced to throw UnsupportedOperationException
  void sleep(); // ❌ same problem
}
// ✅ Good — segregated, clients only depend on what they use
interface Workable  { void work(); }
interface Feedable  { void eat();  }
interface Restable  { void sleep(); }
class HumanWorker implements Workable, Feedable, Restable { /* all three */ }
class RobotWorker  implements Workable                       { /* only work  */ }

// ── D: Dependency Inversion Principle ──────────────────────────────────────
// ❌ Bad — high-level module depends on low-level concrete class
class OrderService {
  private final StripePayment stripe = new StripePayment(); // tightly coupled ❌
  void checkout(Order o) { stripe.charge(o); }
}
// ✅ Good — depend on abstraction; inject via constructor (Spring @Service)
interface PaymentGateway { void charge(Order order); }

@Service
class OrderService {
  private final PaymentGateway gateway; // ← abstraction
  public OrderService(PaymentGateway gateway) { this.gateway = gateway; }
  void checkout(Order o) { gateway.charge(o); }
}
@Service class StripeGateway   implements PaymentGateway { public void charge(Order o) { /* Stripe */ } }
@Service class RazorpayGateway implements PaymentGateway { public void charge(Order o) { /* Razorpay */ } }
// Spring injects correct impl — swap without touching OrderService

Hard

synchronized method: Acquires lock on this (instance) or class object (static).

Q14
Comparable vs Comparator — when to use which?
Easy
Comparable (java.lang): Natural ordering — implemented in the class itself. Single sort sequence.
compareTo(T o) — modifies the class.

Comparator (java.util): External comparator — defined outside. Multiple sort sequences.
compare(T o1, T o2) — doesn't modify the class.

Java 8 Comparator chains:
Comparator.comparing(Employee::getSalary).thenComparing(Employee::getName) 
// ── Full Employee class showing Comparable + multiple Comparators ──────────
public class Employee implements Comparable<Employee> {
  private final String name;
  private final String department;
  private final double salary;
  private final int    age;

  public Employee(String name, String dept, double salary, int age) {
    this.name = name; this.department = dept;
    this.salary = salary; this.age = age;
  }
  // Natural ordering — defined INSIDE class (used by Collections.sort / TreeSet)
  @Override
  public int compareTo(Employee other) {
    return this.name.compareTo(other.name); // natural order = alphabetical by name
  }
  public String getName()       { return name; }
  public String getDepartment() { return department; }
  public double getSalary()     { return salary; }
  public int    getAge()        { return age; }
  @Override public String toString() { return name + "(" + salary + ")"; }
}

public class ComparatorDemo {
  public static void main(String[] args) {
    List<Employee> employees = Arrays.asList(
      new Employee("Zara", "HR",      60000, 28),
      new Employee("Alice", "IT",     95000, 32),
      new Employee("Bob", "IT",       95000, 29),
      new Employee("Charlie", "Finance", 72000, 35)
    );

    // 1. Natural ordering via Comparable (alphabetical by name)
    Collections.sort(employees);
    System.out.println("Natural: " + employees);

    // 2. Comparator — sort by salary descending (external, doesn't touch class)
    Comparator<Employee> bySalaryDesc =
      Comparator.comparingDouble(Employee::getSalary).reversed();
    employees.sort(bySalaryDesc);
    System.out.println("By salary DESC: " + employees);

    // 3. Multi-level sort — salary desc, then name asc (tiebreaker)
    Comparator<Employee> multiSort =
      Comparator.comparingDouble(Employee::getSalary)
                 .reversed()
                 .thenComparing(Employee::getName);
    employees.sort(multiSort);
    System.out.println("Multi-sort (salary DESC, name ASC): " + employees);

    // 4. Comparator.comparing with null-safety
    Comparator<Employee> nullSafe =
      Comparator.comparing(Employee::getDepartment,
                            Comparator.nullsLast(Comparator.naturalOrder()));

    // 5. Group by department, then sort by salary within each dept
    Map<String, List<Employee>> byDept = employees.stream()
      .sorted(Comparator.comparing(Employee::getDepartment)
                        .thenComparingDouble(Employee::getSalary).reversed())
      .collect(Collectors.groupingBy(Employee::getDepartment));

    // 6. TreeMap with custom Comparator (keeps employees sorted)
    TreeSet<Employee> treeSet = new TreeSet<>(
      Comparator.comparingDouble(Employee::getSalary).thenComparing(Employee::getName));
    treeSet.addAll(employees);
    System.out.println("TreeSet (salary asc): " + treeSet);
  }
}

/*
 Key difference summary:
   Comparable  → natural order, ONE definition, inside the class
   Comparator  → custom order, MULTIPLE definitions, outside the class
   Use Comparable when class has a clear natural sort (Integer, String, Date)
   Use Comparator when you need different orderings or can't modify the class
*/

How does synchronized work internally in JVM? (Monitor locks)
Hard
Every Java object has an associated monitor (intrinsic lock). synchronized uses bytecode instructions MONITORENTER and MONITOREXIT.

How it works:
  • Thread attempts to acquire the monitor
  • If free → thread owns it, enters critical section
  • If taken → thread enters BLOCKED state until monitor is released
  • Re-entrant: same thread can acquire its own lock again (counts incremented)

synchronized(this) { // locks object instance }
synchronized(MyClass.class) { // locks class-level (static) }
PitfallTwo synchronized methods on SAME object share one lock — only one thread can run either at a time.

How does ThreadPoolExecutor work? How do you tune it for production?
Hard🏭 Prod
ThreadPoolExecutor key params:
  • corePoolSize: Threads always alive
  • maxPoolSize: Max threads when queue full
  • keepAliveTime: Extra threads beyond core die after this idle time
  • workQueue: Tasks wait here (LinkedBlockingQueue, SynchronousQueue, etc.)
  • RejectedExecutionHandler: Policy when queue and pool are full
Tuning formula:
  • I/O-bound: threads = CPU cores × 2
  • CPU-bound: threads = CPU cores + 1
  • Mixed: threads = cores × (1 + wait/compute ratio)
ProductionMonitor queue depth + rejection rate via metrics. If queue grows, increase corePoolSize or optimize tasks.
import java.util.concurrent.*;

public class ThreadPoolDemo {
  public static void main(String[] args) throws InterruptedException, ExecutionException {

    // Production-grade ThreadPoolExecutor — explicit config
    int cpuCores = Runtime.getRuntime().availableProcessors();
    ThreadPoolExecutor executor = new ThreadPoolExecutor(
      cpuCores,                          // corePoolSize: always alive
      cpuCores * 2,                    // maxPoolSize: max threads when queue full
      60, TimeUnit.SECONDS,            // keepAliveTime: extra threads die after idle
      new LinkedBlockingQueue<>(500),  // bounded queue (avoid OOM from unbounded)
      new ThreadFactory() {           // named threads for easy debugging
        private int count = 0;
        public Thread newThread(Runnable r) {
          return new Thread(r, "worker-" + count++);
        }
      },
      new ThreadPoolExecutor.CallerRunsPolicy() // rejection: caller runs task itself
      // Other policies: AbortPolicy (throws), DiscardPolicy (silently drops),
      //                 DiscardOldestPolicy (drops oldest pending task)
    );

    // Submit tasks and get Future results
    Future<String> future = executor.submit(() -> {
      Thread.sleep(200);
      return "Result from " + Thread.currentThread().getName();
    });

    try {
      String result = future.get(1, TimeUnit.SECONDS); // timeout prevents hung futures
      System.out.println(result);
    } catch (TimeoutException e) {
      future.cancel(true); // interrupt running task
    }

    // Monitor pool health (wire to metrics in prod)
    System.out.printf("Active=%d Queue=%d Completed=%d%n",
      executor.getActiveCount(),
      executor.getQueue().size(),
      executor.getCompletedTaskCount());

    // Graceful shutdown — wait up to 30s for tasks to finish
    executor.shutdown();
    if (!executor.awaitTermination(30, TimeUnit.SECONDS)) {
      executor.shutdownNow(); // force interrupt remaining tasks
    }
  }

import java.util.concurrent.*;
import java.util.concurrent.locks.*;

public class ReentrantLockDemo {
  private final ReentrantLock lock = new ReentrantLock(true); // fair=true
  private int balance = 1000;

  // ✅ Standard ReentrantLock usage — always unlock in finally
  public void debit(int amount) {
    lock.lock(); // blocks until lock acquired
    try {
      if (balance >= amount) {
        balance -= amount;
        System.out.println("Debited " + amount + ", balance=" + balance);
      }
    } finally {
      lock.unlock(); // ALWAYS in finally — exception safe
    }
  }

  // ✅ tryLock — non-blocking, avoids deadlock
  public boolean tryDebit(int amount) throws InterruptedException {
    if (lock.tryLock(100, TimeUnit.MILLISECONDS)) { // wait up to 100ms
      try {
        balance -= amount;
        return true;
      } finally { lock.unlock(); }
    }
    System.out.println("Could not acquire lock — skipping");
    return false; // did not wait — no deadlock risk
  }

  // ✅ ReentrantLock with Condition — finer-grained than wait/notify
  private final Condition sufficientFunds = lock.newCondition();

  public void withdraw(int amount) throws InterruptedException {
    lock.lock();
    try {
      while (balance < amount) {
        System.out.println("Insufficient funds, waiting...");
        sufficientFunds.await(); // release lock and wait for signal
      }
      balance -= amount;
    } finally { lock.unlock(); }
  }

  public void deposit(int amount) {
    lock.lock();
    try {
      balance += amount;
      sufficientFunds.signalAll(); // wake up threads waiting for funds
    } finally { lock.unlock(); }
  }

  // When to use ReentrantLock over synchronized:
  // ✅ Need tryLock() to avoid deadlock
  // ✅ Need lockInterruptibly() — cancellable waiting
  // ✅ Need multiple Condition objects (different wait sets)
  // ✅ Need fairness (FIFO ordering)
  // ✅ Virtual threads with blocking I/O (synchronized pins carrier thread)
  // ❌ synchronized is fine for simple mutual exclusion — less error-prone

// ── CountDownLatch — main waits for all workers to finish ──────────────────
public class CountDownLatchDemo {
  public static void main(String[] args) throws InterruptedException {
    int workerCount = 5;
    CountDownLatch latch = new CountDownLatch(workerCount); // count = 5
    ExecutorService pool = Executors.newFixedThreadPool(workerCount);

    for (int i = 1; i <= workerCount; i++) {
      final int id = i;
      pool.submit(() -> {
        try {
          System.out.println("Worker-" + id + " started");
          Thread.sleep(500 * id); // simulate work
          System.out.println("Worker-" + id + " done");
        } catch (InterruptedException e) {
          Thread.currentThread().interrupt();
        } finally {
          latch.countDown(); // decrement count — ALWAYS in finally
        }
      });
    }

    System.out.println("Main thread waiting for all workers...");
    latch.await(30, TimeUnit.SECONDS); // waits until count == 0 (or timeout)
    System.out.println("All workers finished — aggregating results");
    pool.shutdown();
  }
}

public class CyclicBarrierDemo {
  public static void main(String[] args) throws InterruptedException {
    int parties = 3;
    // Optional barrier action — runs once when all parties arrive
    CyclicBarrier barrier = new CyclicBarrier(parties,
        () -> System.out.println("=== All threads reached checkpoint — proceeding ==="));

    ExecutorService pool = Executors.newFixedThreadPool(parties);

    for (int phase = 1; phase <= 2; phase++) {  // 2 rounds (barrier is reusable!)
      final int p = phase;
      for (int i = 1; i <= parties; i++) {
        final int id = i;
        pool.submit(() -> {
          try {
            System.out.println("Thread-" + id + " phase-" + p + ": working");
            Thread.sleep(200 * id);
            System.out.println("Thread-" + id + " reached barrier");
            barrier.await(); // wait here until all 3 threads arrive
            // all 3 threads continue only after all arrive
            System.out.println("Thread-" + id + " phase-" + p + ": continuing");
          } catch (Exception e) { Thread.currentThread().interrupt(); }
        });
      }
      Thread.sleep(2000); // let phase complete before starting next
    }
    pool.shutdown();
  }
}

Hard🔥 Hot

What is the N+1 problem in JPA? How do you detect and fix it?
Hard🔥 Hot
Problem: Loading 1 list of N entities + N additional queries to load related entities.
// Fetches 1 query for orders, then N queries for each order's customer

Detection: Enable spring.jpa.show-sql=true + Hibernate statistics. Use p6spy or datasource-proxy.

Fixes:
  • @EntityGraph — eager join fetch at query level
  • JPQL JOIN FETCH: SELECT o FROM Order o JOIN FETCH o.customer
  • Batch loading: @BatchSize(size = 20)
  • DTO projections: fetch only needed columns
// ❌ N+1 PROBLEM — Entity with LAZY fetch

  private Customer customer;
  @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)

}

// ❌ This triggers N+1: 1 query for orders + N queries for each customer

JPA-3
Explain Hibernate First-Level and Second-Level cache. When does it hurt?
Hard
First-Level Cache (Session Cache): Per-session, always enabled. Same entity loaded twice in one transaction → single DB hit. Cleared on session.close().

Second-Level Cache (Shared Cache): Cross-session cache. Requires explicit config (Ehcache, Redis, Caffeine). Configured per entity with @Cache.

When it hurts:
  • Stale data: another service updates DB directly — cache not invalidated
  • Memory pressure: too many cached entities
  • Large associations being eagerly cached
GotchaDirect JPQL/native updates bypass first-level cache — entity state in memory becomes stale.

What is the Saga pattern? Choreography vs Orchestration?
Hard🔥 Hot

How does Kafka ensure message ordering and exactly-once delivery?
Hard
Ordering: Guaranteed within a partition, not across partitions. Use same key → same partition.

Message Durability — acks:
  • acks=0: No acknowledgment — fastest, possible loss
  • acks=1: Leader acknowledges — leader failure = loss
  • acks=all: All ISR replicas acknowledge — strongest guarantee
Exactly-once: Enable idempotent producer (enable.idempotence=true) + transactional API. Prevents duplicates from retries. Combined with consumer read_committed isolation.

Consumer crash after consume, before commit: Message re-delivered on restart (at-least-once). Use idempotent processing to handle duplicates.

How does Redis caching work? Cache-Aside vs Write-Through?
Medium

Kadane's Algorithm — Maximum Subarray Sum
public class MaxSubarray {
  public static int maxSubarraySum(int[] nums) {
    int maxSoFar = nums[0];
    int currentMax = nums[0];

    for(int i = 1; i < nums.length; i++) {
      // Either extend the current subarray, or start fresh
      currentMax = Math.max(nums[i], currentMax + nums[i]);
      maxSoFar = Math.max(maxSoFar, currentMax);
    }
    return maxSoFar;
  }

  // Variant: return start and end indices of the subarray
  public static int[] maxSubarrayWithBounds(int[] nums) {
    int start = 0, end = 0, tempStart = 0;
    int maxSoFar = nums[0], currentMax = nums[0];
    for(int i = 1; i < nums.length; i++) {
      if(nums[i] > currentMax + nums[i]) {
        currentMax = nums[i]; tempStart = i;
      } else { currentMax += nums[i]; }
      if(currentMax > maxSoFar) {
        maxSoFar = currentMax; start = tempStart; end = i;
      }
    }
    return new int[]{maxSoFar, start, end};
  }

  public static void main(String[] args) {
    int[] arr = {-2, 1, -3, 4, -1, 2, 1, -5, 4};
    System.out.println(maxSubarraySum(arr)); // 6 (subarray [4,-1,2,1])
  }
}
// Time: O(n) | Space: O(1)

Valid Parentheses — check if bracket string is balanced ({[()]})
import java.util.*;

public class ValidParentheses {
  public static boolean isValid(String s) {
    Deque<Character> stack = new ArrayDeque<>();
    Map<Character, Character> pairs = Map.of(
      ')', '(',
      ']', '[',
      '}', '{'
    );

    for(char c : s.toCharArray()) {
      if("({[".indexOf(c) != -1) {
        stack.push(c);  // opening bracket — push
      } else {
        // closing bracket — must match top of stack
        if(stack.isEmpty() || stack.pop() != pairs.get(c)) {
          return false;
        }
      }
    }
    return stack.isEmpty(); // valid only if all brackets matched
  }

  public static void main(String[] args) {
    System.out.println(isValid("({[]})"));  // true
    System.out.println(isValid("([)]"));    // false
    System.out.println(isValid("{[]}"));    // true
  }
}

Reverse a Linked List — iterative and recursive
class ListNode {
  int val;
  ListNode next;
  ListNode(int v) { val = v; }
}

public class ReverseLinkedList {
  // Iterative — O(n) time, O(1) space
  public static ListNode reverseIterative(ListNode head) {
    ListNode prev = null, curr = head;
    while(curr != null) {
      ListNode next = curr.next; // save next
      curr.next = prev;          // reverse pointer
      prev = curr;               // advance prev
      curr = next;               // advance curr
    }
    return prev; // new head
  }

  // Recursive — O(n) time, O(n) stack space
  public static ListNode reverseRecursive(ListNode head) {
    if(head == null || head.next == null) return head;
    ListNode newHead = reverseRecursive(head.next);
    head.next.next = head; // reverse pointer
    head.next = null;     // break old link
    return newHead;
  }

  public static void main(String[] args) {
    // Build: 1 → 2 → 3 → 4 → 5
    ListNode head = new ListNode(1);
    head.next = new ListNode(2);
    head.next.next = new ListNode(3);
    ListNode rev = reverseIterative(head);
    // Result: 3 → 2 → 1
  }
}

Longest Substring Without Repeating Characters
public class LongestSubstring {
  public static int lengthOfLongestSubstring(String s) {
    Map<Character, Integer> lastSeen = new HashMap<>();
    int maxLen = 0;
    int left = 0; // sliding window left boundary

    for(int right = 0; right < s.length(); right++) {
      char c = s.charAt(right);
      if(lastSeen.containsKey(c) && lastSeen.get(c) >= left) {
        left = lastSeen.get(c) + 1; // shrink window past duplicate
      }
      lastSeen.put(c, right);
      maxLen = Math.max(maxLen, right - left + 1);
    }
    return maxLen;
  }

  public static void main(String[] args) {
    System.out.println(lengthOfLongestSubstring("abcabcbb")); // 3 ("abc")
    System.out.println(lengthOfLongestSubstring("pwwkew"));   // 3 ("wke")
    System.out.println(lengthOfLongestSubstring("bbbbb"));    // 1
  }
}
// Time: O(n) | Space: O(min(m,n)) where m=charset size

3 threads must print numbers 1–10 in sequence (1,2,3,4... with each thread taking turns). Implement this.
Hard🔥 Hot
public class SequentialThreads {
  private static int current = 1;
  private static int turn = 0;        // 0→T1, 1→T2, 2→T3
  private static final int MAX = 10;
  private static final Object lock = new Object();

  static class PrintThread implements Runnable {
    private final int threadId;
    PrintThread(int id) { this.threadId = id; }

    @Override
    public void run() {
      while(true) {
        synchronized(lock) {
          while(turn != threadId || current > MAX) {
            if(current > MAX) return;
            try { lock.wait(); } catch(InterruptedException e) { return; }
          }
          System.out.println("Thread-" + (threadId+1) + ": " + current);
          current++;
          turn = (turn + 1) % 3; // pass turn to next thread
          lock.notifyAll();
        }
      }
    }
  }

  public static void main(String[] args) throws InterruptedException {
    Thread[] threads = {
      new Thread(new PrintThread(0), "T1"),
      new Thread(new PrintThread(1), "T2"),
      new Thread(new PrintThread(2), "T3")
    };
    for(Thread t : threads) t.start();
    for(Thread t : threads) t.join();
  }
}
// Output: Thread-1: 1, Thread-2: 2, Thread-3: 3, Thread-1: 4 ...

import java.util.*;

public class DuplicatesToEnd {
  public static int[] solve(int[] arr) {
    LinkedHashSet<Integer> unique = new LinkedHashSet<>();
    List<Integer> duplicates = new ArrayList<>();

    for(int x : arr) {
      if(!unique.add(x)) {   // add() returns false if already present
        duplicates.add(x);
      }
    }

    List<Integer> result = new ArrayList<>(unique);
    result.addAll(duplicates);
    return result.stream().mapToInt(Integer::intValue).toArray();
  }

  public static void main(String[] args) {
    int[] arr = {1, 2, 1, 3, 2, 4};
    System.out.println(Arrays.toString(solve(arr)));
    // Output: [1, 2, 3, 4, 1, 2]
  }
}

Find maximum sum subarray of size K (Sliding Window)
Medium🔥 Hot
public class SlidingWindowMaxSum {
  public static int maxSum(int[] arr, int k) {
    if(arr.length < k) throw new IllegalArgumentException("k > array length");

    // Calculate first window sum
    int windowSum = 0;
    for(int i = 0; i < k; i++) windowSum += arr[i];
    int maxSum = windowSum;

    // Slide window: add right element, remove left element
    for(int i = k; i < arr.length; i++) {
      windowSum += arr[i] - arr[i - k]; // O(1) — no need to re-sum
      maxSum = Math.max(maxSum, windowSum);
    }
    return maxSum;
  }

  public static void main(String[] args) {
    int[] arr = {2, 1, 5, 1, 3, 2};
    System.out.println(maxSum(arr, 3)); // Output: 9 (5+1+3)
  }
}
Time: O(n) — one pass through array | Space: O(1). Much better than brute force O(n×k).

Implement retry with exponential backoff in Java
Medium🏭 Prod🔥 Hot
import java.util.function.*;

public class RetryUtil {

  public static <T> T retryWithBackoff(
      Supplier<T> operation,
      int maxRetries,
      long initialDelayMs,
      double multiplier) {

    long delay = initialDelayMs;
    for(int attempt = 1; attempt <= maxRetries; attempt++) {
      try {
        return operation.get();
      } catch(Exception e) {
        if(attempt == maxRetries) throw new RuntimeException("Max retries reached", e);
        System.err.printf("Attempt %d failed: %s. Retrying in %dms%n",
          attempt, e.getMessage(), delay);
        try { Thread.sleep(delay); } catch(InterruptedException ie) { Thread.currentThread().interrupt(); throw new RuntimeException(ie); }
        delay = (long)(delay * multiplier); // exponential increase
      }
    }
    throw new RuntimeException("Should not reach here");
  }

  public static void main(String[] args) {
    String result = retryWithBackoff(
      () -> callExternalService(), // operation to retry
      3,      // max retries
      200L,   // initial delay 200ms
      2.0    // double delay each time: 200 → 400 → 800ms
    );
  }

  // ✅ In production, use Resilience4j:
  // @Retry(name = "extService", fallbackMethod = "fallback")
  // resilience4j.retry.instances.extService.maxAttempts=3
  // resilience4j.retry.instances.extService.waitDuration=200ms
  // resilience4j.retry.instances.extService.enableExponentialBackoff=true
  // resilience4j.retry.instances.extService.exponentialBackoffMultiplier=2
}
